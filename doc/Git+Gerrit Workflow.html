mark7's Git+Gerrit Maginatics Workflow

Basics

Creating a new patch

Here's a basic summary of how I use git. I think that it's pretty straightforward, and I believe that most people here use git in roughly this fashion. I strongly advise against committing patches into your local master, as it creates all sorts of problems for you. This workflow will avoid committing patches into local master.

You start out with a git repo containing one local branch, master. This tracks origin/master (I use an asterisk to show that this is the active branch) 

  1 -> 2 -> 3 -> 4 [*master]

When you want to make changes, you run: 

  git checkout -b wb-foo

This creates a new branch that is identical to master and points at the same commit at the moment: 

  1 -> 2 -> 3 -> 4 [master]
                   [*wb-foo]

You make your changes on this branch. Most of the time, I only need one commit, though some people make many commits and squash them down into one. Giving the “one-commit” process, you have wb-foo checked out. You'll make changes, stage them for a commit: 

  git add -u

And then commit them: 

  git commit

These become #5 here: 

  1 -> 2 -> 3 -> 4 [master]
                   \ 5 [*wb-foo]

I never commit anything directly into local master. Local master just mirrors what lives up on the server in origin/master.

When I want to make more changes, while on the wb-foo branch, I use 

  git add -u

to stage the changes and 

  git commit --amend

to modify the existing commit: 

  1 -> 2 -> 3 -> 4 [master]
                   \ 6 [*wb-foo]

When I'm ready to push the patch out to Gerrit, I run: 

  git push origin HEAD:refs/for/master

Updating master

Sometimes, I need to pull in a newer version of master. If I do that I'll first checkout master to switch from wb-foo to master: 

  git checkout master

Then I will “git pull”. This will pull down new changes from origin, then tack them onto the end of my local master branch. This operation cannot fail, because I've not committed anything into my local master branch. I'm just mirroring what I know is on the origin server: 

  git pull

Now I have this: 

  1 -> 2 -> 3 -> 4 -> 6 -> 7 [*master]
                 \ 5 [wb-foo]

Then I will rebase wb-foo. This tries to take wb-foo from being attached to #4 and attaching it to #7, the current HEAD on master. This will involve merging the changes from #6 to #7 on master into wb-foo: 

  git checkout wb-foo

Giving 

  1 -> 2 -> 3 -> 4 -> 6 -> 7 [master]
                 \ 5 [*wb-foo]

Then 

  git rebase master

If there are conflicts (which could happen if my change, #5, conflicts with some other change #6 or #7, I may wind up with a conflict. If I do 

  git show

I can see the list of files that conflict, since git will not have added these. I can either give up on the rebase (if the conflict is too complicated for me to resolve at the moment): 

  git rebase --abort

Or fix the conflicting changes, and then stage them for merging: 

  git add <conflicting file>

And the continue with the rebase: 

  git rebase --continue

At the end, I will be on wb-foo, and it will be rebased onto HEAD on master: 

  1 -> 2 -> 3 -> 4 -> 6 -> 7 [master]
                           \ 8 [*wb-foo]

I can now push it up to Gerrit: 

  git push origin HEAD:refs/for/master

If I want to work on multiple patches simultaneously, I just create more working branches attached to master.

That covers the majority of the work I do. There are a few other variants that can be useful:

Large patches

For a large patch, I may add multiple commits onto wb-foo with 

  git add -u

and 

  git commit

for each commit. 

  1 -> 2 -> 3 -> 4 [master]
                  \ 5 -> 6 -> 7 [*wb-foo]

When I'm done, I squish things down to one patch and push it up, marking the commits to be “squashed” into the previous commit as “squash” (here, #6 and #7): 

  git rebase -i master

resulting in: 

  1 -> 2 -> 3 -> 4 [master]
                  \ 8 [*wb-foo]

Dependent patches

I may also have a series of dependent patches, which I've not had to do much: 

  1-> 2 -> 3 -> 4 -> 7 -> 8 [master]
                 \ 5 [wb-foo]
                    \ 6 [*wb-bar]

Here, if I pull in new versions to master: I need to rebase both wb-foo onto wb-master: 

  git checkout wb-foo
  1-> 2 -> 3 -> 4 -> 7 -> 8 [master]
                 \ 5 [*wb-foo]
                    \ 6 [wb-bar]
  git rebase wb-foo master
  1-> 2 -> 3 -> 4 -> 7 -> 8 [master]
                 \ 5       \9 [*wb-foo]
                    \ 6 [wb-bar]

And rebase wb-bar onto the new version of wb-foo: 

  git checkout wb-bar

Giving 

  1-> 2 -> 3 -> 4 -> 7 -> 8 [master]
                 \ 5       \9 [wb-foo]
                    \ 6 [*wb-bar]

Then 

  git rebase wb-foo

Giving 

  1-> 2 -> 3 -> 4 -> 7 -> 8 [master]
                           \9 [wb-foo]
                            \ 10 [*wb-bar]

I think that that covers the important stuff.

Extras

Extracting patches from a branch and applying them elsewhere

The most-useful non-crucial stuff I use much is probably: 

  git cherry-pick

This basically makes a diff out of a commit, patches it onto another branch, and creates a new commit. For example, if I have: 

  1 -> 2 -> 3 -> 4 [master]
       |          \ 5 [wb-foo]
        \ 6 [wb-bar]

I could grab the wb-bar commit, convert it into a diff, and insert it onto the wb-foo branch: 

 git checkout wb-foo
 git cherry-pick wb-bar
  1 -> 2 -> 3 -> 4 [master]
       |          \ 5 -> 7 [wb-foo]
        \ 6 [wb-bar]

Where the new commit #7 contains the same changes that were in commit #6.

Tracking what you've done

Git reflog shows a log of the last git operations performed. E.g.: 

  fe2caa0 HEAD@{0}: commit (amend): Avoid crashing on rapid chandle
  destruction
  aaf0b58 HEAD@{1}: commit: Avoid crashing on rapid chandle destruction
  e5fff4d HEAD@{2}: checkout: moving from master to wb-9937

I can reach any of these states by simply checking out the hashref at the beginning of the line. Even if there's no branch or tag or anything that references one of these states, it will linger for two weeks until git garbage-collects it.

No matter how tangled up you get, you can see what happened by looking at git reflog.

Removing commits from local branches

Git reset is important mostly for fixing errors. A branch is just a “label” that points at a commit. It's like a tag. For local branches, it's possible to move the label to point at something else. 

  1 -> 2 -> 3 -> 4 [master]
                 \ 5 [*wb-foo]

And you make changes, git add -u, and git commit –amend wb-foo to “replace” commit #5 with commit #6: 

  1 -> 2 -> 3 -> 4 [master]
                 \ 6 [*wb-foo]

Then you realize that you didn't want to have overwritten #5. Just git reflog, get the hashref associated with commit #5, and then git –reset wb-foo to point at commit #5 again: 

  git reset --hard aaf0b58

And voila: 

  1 -> 2 -> 3 -> 4 [master]
                 \ 5 [*wb-foo]

This is also handy for “dropping” the last commit on a local branch by making a branch point at HEAD~ (one *before* HEAD): 

  1 -> 2 -> 3 -> 4 [master]
                 \ 5 -> 6 [*wb-foo]

Then: 

  git reset --hard HEAD~

Giving: 

  1 -> 2 -> 3 -> 4 [master]
                 \ 5 [*wb-foo]
